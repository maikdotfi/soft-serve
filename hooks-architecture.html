🔧 Soft Serve: Git Hook Subprocess Architecture

Why every git push spawns a fresh soft hook process — and what that means for CI, backups, and webhooks.

Client soft serve daemon Hook subprocess CI work Backup work

⚡ The core constraint

Git's server-side hooks (pre-receive, update, post-receive, post-update) are always spawned as standalone child processes by git-receive-pack. Git exec()s a script — it does not call a function inside the running server. Soft Serve has no choice: every hook must be a subprocess.

Key design property: The soft hook subprocess is a complete, self-contained Go binary invocation. Each invocation opens the database, wires up the backend + backup + CI services, performs its work, and exits. It is not an RPC call to the soft serve daemon — it is an independent process.

🎯 Why a subprocess — not an RPC or socket

Other Git servers (Gitea, GitLab) use the same pattern: the hook script contacts the running server via HTTP/socket. Soft Serve re-initializes the entire backend in a new process instead. Why?

▶ Simplicity — no extra server surface

No need for an admin RPC endpoint, authentication for hooks, or connection management. The hook subprocess just opens the database directly (same SQLite/Postgres file the daemon uses) and does its work. This means hooks work even if the HTTP/SSH server is down or restarting.

▶ Transactional consistency — each hook is self-contained

The subprocess opens its own DB connection, reads current state, performs validation/work, and commits within a single transaction scope. There is no shared mutable state with the daemon process to get wrong.

▶ No long-lived connection to manage

No WebSocket, no gRPC stream, no HTTP keep-alive between daemon and hook. The subprocess is ephemeral: start, work, report exit code to Git, die. If the daemon restarts mid-push, the hook subprocess is unaffected.

▶ Downside: repeated initialization cost

Each hook subprocess pays the full startup cost: Go runtime init, DB connection open, service wiring. On a push that updates 3 refs, this means 3× pre-receive + 3× update + 3× post-receive = up to 9 subprocess invocations. Soft Serve mitigates this by using SQLite (fast local file, connection in ~1ms) and keeping service wiring lightweight, but it's still a non-zero overhead per push.

The project AGENTS.md mentions this as a potential future improvement — see the "TODO: run this async" comment in pkg/backend/hooks.go and the note about "an RPC server to communicate with the hook process".

🔁 How CI fits into this

The CI subsystem runs along two separate paths. Only one requires a subprocess.

📥 Path A: Push-triggered (in hook subprocess)

Must run in a subprocess because it lives inside Git's hook chain.

When a user pushes, two CI operations execute inside the hook subprocess:

Operation Runs in What it does
CIPreReceive hook pre-receive Parses .soft-serve/workflows/*.yml from the incoming commit tree. If the YAML is unparseable, writes error to stderr and exits non-zero → push is rejected before any refs are updated.
CIPostReceive hook post-receive Reconciles the stored Workflow set with what's now at HEAD (the push was already accepted). Upserts new workflows, deletes removed ones. Failures are logged but do not affect the client.
Code path: cmd/soft/hook/hook.go lines 83–89: 
case hooks.PreReceiveHook:
    if err := hks.CIPreReceive(ctx, repoName, opts, stderr); err != nil {
        return err   // non-zero exit → git rejects the push
    }
    hks.PreReceive(ctx, stdout, stderr, repoName, opts)
case hooks.PostReceiveHook:
    hks.PostReceive(ctx, stdout, stderr, repoName, opts)
    hks.CIPostReceive(ctx, repoName, opts)

📤 Path B: Webhook-triggered (in daemon process)

Runs in the soft serve daemon — no subprocess needed.

When non-push events fire (branch/tag creation, collaborator changes, repo visibility changes), the in-process webhook handler fans them out into pending CI runs. This is registered only in serve.go, not in the hook subprocess:

💾 Backup also splits across both paths

Trigger Subprocess? Details
Push to default branch Yes (post-receive hook) PostReceive calls backup.HandlePushToDefaultBranch. The subprocess must call svc.Wait() before exiting (see PersistentPostRunE in cmd/soft/hook/hook.go) or the S3 upload goroutine gets killed mid-flight.
Scheduled (cron) backup No (daemon) CreateDefaultBackupSchedule is only called in serve.go. The long-running daemon fires backups on a timer.
Gotcha: Without WireOptionalServices in the hook subprocess's PersistentPreRunE, b.backup == nil and push-triggered backups silently no-op with nothing logged. This is explicitly documented in cmd/hook/hook.go and cmd/services.go.

📋 Complete picture

Operation Subprocess? Which process Why not in daemon?
Pre-receive CI validation Yes soft hook pre-receive Git spawns pre-receive as a child process; must exit non-zero to reject push
Pre-receive webhook (branch/tag) Yes soft hook pre-receive Same Git architecture constraint
Update hook (per-ref webhooks) Yes soft hook update Same — Git passes ref/old/new as args to a child process
Post-receive CI workflow sync Yes soft hook post-receive Push already accepted; runs in post-receive hook chain
Post-receive backup trigger Yes soft hook post-receive Same — part of post-receive hook chain
Webhook → CI run creation No soft serve daemon In-process event handler registered via SetFiredEventHandler
Scheduled backups No soft serve daemon Needs long-lived cron timer

📎 The generated hook scripts

pkg/hooks/gen.go writes two files per hook into each bare repo's hooks/ directory:

▶ repo.git/hooks/pre-receive — dispatcher script 
#!/usr/bin/env bash
# AUTO GENERATED BY SOFT SERVE, DO NOT MODIFY
data=$(cat)
exitcodes=""
hookname=$(basename $0)
GIT_DIR=${GIT_DIR:-$(dirname $0)/..}
for hook in ${GIT_DIR}/hooks/${hookname}.d/*; do
  test -x "${hook}" && test -f "${hook}" || continue
  echo "${data}" | "${hook}" "$@"
  exitcodes="${exitcodes} $?"
done
for i in ${exitcodes}; do
  [ ${i} -eq 0 ] || exit ${i}
done

Reads stdin, iterates pre-receive.d/*, collects exit codes, exits on first failure. This allows third-party hooks to coexist with Soft Serve's own hook.

▶ repo.git/hooks/pre-receive.d/soft-serve — the actual hook 
#!/usr/bin/env bash
# AUTO GENERATED BY SOFT SERVE, DO NOT MODIFY
if [ -z "$SOFT_SERVE_REPO_NAME" ]; then
    echo "Warning: SOFT_SERVE_REPO_NAME not defined. Skipping hooks."
    exit 0
fi
"${SOFT_SERVE_BIN_PATH}" hook pre-receive

Invokes the soft binary's hook pre-receive subcommand. SOFT_SERVE_BIN_PATH and SOFT_SERVE_REPO_NAME are set by the SSH/HTTP transport layer before invoking git-receive-pack.
